tecs数据库Schema版本方案参见文档：ZXTULIP(TECS)_V01.01.10/03 设计阶段/TECS数据库Schema版本控制.docx

所有的*.sql文件必须是Unicode格式！！！

这里列出该文档中针对数据库升级的7个游戏规则：

规则一：表和视图的创建脚本需要分离（xx_baseline.sql中只能存放create table语句，create view必须放到xx_view.sql中）

规则二：基线Schema提交之后不再允许修改（xx_baseline.sql一旦提交到svn，永远都是只读的！！！）

规则三：每个数据库中创建一个_schema_log表以记录Schema升级过程（这个是安装工具自己去自动创建的，开发人员可以不用关心）

规则四：表结构的每次变更必须创建独立的脚本来完成（如果需要增加、删除或修改一个表，请新建一个update脚本来完成，注意版本号要++）

规则五：update脚本提交之后不再允许修改（update_*.sql一旦提交到svn，永远都是只读的！！！）

规则六：视图的定义发生变化时需要创建一个版本号升级脚本（视图变化时，请直接修改xx_view.sql，然后和规则四一样创建一个update脚本，该脚本中只要负责升级版本号，同规则五一样，视图的update脚本提交之后也不能再修改！）

规则七：当updates目录出现了bugfix.*文件之后，就必须进入补丁模式维护，即不允许再增加任何的update脚本了，而是必须增加bugfix/reverse脚本。

这7个规则是开发人员进行数据库Schema变更时必须遵守的，不容有任何违规行为！！！

关于规则的详细介绍和解释请参见：《TECS数据库Schema版本控制.docx》

“数据库Schema变更”包括：
* 增加一个新的表
* 增加一个新的视图
* 给已有的表增加新的字段
* 从已有的表中删除字段
* 给已有的表增加索引
* 修改已有的视图
* ……
